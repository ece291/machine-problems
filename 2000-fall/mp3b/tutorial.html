How to Do this MP

I could have a huge reference section, and then a huge code description section, but I decided to do a little bit at a time in a tutorial format.  "Here's how this works.  Now here's how you write the code.  Now write the code."  It's probably not worth reading over the whole thing before you start.  Just sit down at a computer and go through it step by step.  It's going to some time to go through this, but you're going to learn everything the right way with no "black boxes" or mysterious files given to you by the TAs.  If a file does seem mysterious, type it in instead of blindly pasting it in.  Look at the references.  Ask the guy (or gal) next to you.  Hopefully you'll learn not only 32-bit protected mode, but a little about how assemblers, compilers, and linkers work, learn a little about Makefiles, and learn a little about graphics. So let's begin where every good tutorial should begin -- at the end.

The Goal

The Goal is to learn the basics of protected mode and NASM to prepare you for MP4, where you'll be using the tools and concepts you learn in this part of MP3 to make a cool graphics program in protected mode using a high-resolution, high-color graphics mode.  If you have any questions or comments about this part of the MP, or if something is unclear, incomplete, or just plain wrong, e-mail me at pljohnsn@uiuc.edu.

Protected Mode and Your Final Project

Advantages

  • Better memory management
    • Can allocate huge (multimegabyte) buffers for huge pictures.
    • Don't have to deal with stupid segment limitations.
  • Better instructions
    • Take full advantage of advanced instructions.
    • Reference memory with any register or combination.
  • High color, high resolution graphics (you'll learn this in MP4)
    • Your final projects will look much nicer.
    • Palettes used to be a huge pain.
  • Better tools
    • NASM is much cleaner and more consistent than MASM.
    • NASM uses standard formats and is compatible with standard GNU tools.
  • The Library
    • We give you libraries with source code for some pretty cool stuff.

Disadvantages

  • You're not as used to its beauty.
  • Doing hardware stuff will be harder.
  • Doing sound will be harder (but it's been done in the past, and we're working to make it easier).
  • Debugging will be harder.
